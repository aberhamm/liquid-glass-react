## Requirements

The refraction effect needs a displacement map: an image whose R channel encodes
X displacement and B channel encodes Y displacement, fed into the SVG
`feDisplacementMap`. This plan builds the generator that produces those maps ‚Äî
the `shader` mode generates one at runtime from a rounded-rectangle signed
distance field, and the `standard`/`polar`/`prominent` modes return precomputed
maps. Because we are NOT forking, the SDF math is reimplemented from first
principles (rounded-rect SDF + smoothstep), and the static maps are generated by
our own code rather than copied from upstream's base64 blobs. The "user" is plan
004's refraction component, which calls `getDisplacementMap(mode, width, height)`.

**Acceptance criteria:**

- [ ] `src/displacement.ts` exports `getDisplacementMap(mode: Exclude<
      DisplacementMode, 'turbulence'>, width: number, height: number): string`
      returning a `data:` URL. The type EXCLUDES `'turbulence'` (it is procedural,
      owned by 004) so the call contract is unambiguous ‚Äî `getDisplacementMap` is
      never called for turbulence; 004 routes that mode to `feTurbulence` first.
- [ ] `shader` mode runs a per-pixel rounded-rect SDF + smoothstep on an offscreen
      canvas and encodes displacement as R=X (xChannelSelector) and B=Y
      (yChannelSelector), A=255, then `canvas.toDataURL()`. (G is unused by
      `feDisplacementMap`; leave it 0 or mirror B ‚Äî it does not affect output.)
- [ ] `standard`, `polar`, `prominent` modes return deterministic precomputed
      maps generated by our own helper functions (documented, not copied blobs).
      Each mode has a DISTINCT, documented visual character (eng-review 2026-06-13)
      ‚Äî e.g. `standard` = rounded-rect lens edge (the SDF band); `polar` =
      radial/angular displacement from center; `prominent` = wider, higher-contrast
      edge band (stronger refraction). The descriptions may stay loose, but the four
      map modes must be genuinely different generators, not aliases.
- [ ] A unit test asserts the four map modes are DISTINCT: at one fixed
      `(width,height)`, `getDisplacementMap('standard')`, `'polar'`, `'prominent'`,
      and `'shader'` each return a different data-URL string (no two equal). This
      guards against the "5 advertised modes, 2 real" defect ‚Äî every mode in the
      public `DisplacementMode` union must actually exist and differ.
- [ ] A pure `roundedRectSDF(x,y,w,h,r)` and `smoothStep(a,b,t)` are exported and
      unit-tested against known analytic values (corner, edge, center).
- [ ] Generation is guarded for SSR/no-canvas environments: returns an empty/1x1
      transparent data-URL instead of throwing.
- [ ] Generated maps are cached by `(mode,width,height)` so repeated calls don't
      re-render. **Bounded + quantized** (eng-review 2026-06-13): the cache is a real
      LRU with a hard cap (e.g. 32 entries) ‚Äî never an unbounded `Map` ‚Äî and `width`/
      `height` are quantized to a grid (round to nearest 8‚Äì16px) BEFORE keying and
      generating, so near-identical sizes during a resize share one map. This bounds
      memory and collapses resize-storm regeneration to a handful of renders while
      keeping aspect-correct rounded corners. A unit test asserts (a) eviction past
      the cap and (b) that two sizes within one grid step return the same cached
      data-URL (one render, not two).

## Design

The SDF is the standard rounded-rectangle distance field; displacement at each
pixel is the gradient of a smoothstep band near the edge, normalized by the max
displacement and biased to 0.5 (neutral) so 128 = no shift. Encode R=X and B=Y to
match the `xChannelSelector="R" yChannelSelector="B"` the SVG filter (004) uses;
G is unused and may be left 0 or mirror B.

**Files expected to change:**

- `src/displacement.ts`: `getDisplacementMap`, `roundedRectSDF`, `smoothStep`,
  internal canvas renderer, the precomputed-map generators, and a bounded LRU cache
  (hard cap + dimension quantization ‚Äî see acceptance).
- `src/displacement.test.ts`: analytic SDF/smoothstep assertions, encoding range
  checks (all bytes 0‚Äì255, center ‚âà 128), data-URL prefix checks, cache behavior,
  SSR no-canvas path.
- `src/index.ts`: optionally export `getDisplacementMap` for advanced consumers.

**Bundle-size discipline:** the `standard`/`polar`/`prominent` maps are generated
at RUNTIME (canvas ‚Üí data-URL, cached), NOT shipped as large inline base64 blobs
in the JS bundle ‚Äî a single prebaked displacement map is ~10‚Äì15KB and several
would dominate the package size (this is exactly TODO T2's concern). If a future
optimization wants prebaked maps, emit them as separate build-time assets and
prefer WEBP over PNG/JPEG; never inline base64 maps into the bundle by default.

**Testing approach:** unit-only ‚Äî pure math + canvas encoding, tested with Vitest.
Canvas in jsdom is limited; gate canvas-dependent assertions behind a capability
check and assert the math/encoding directly where canvas is unavailable.

**Out of scope:** the SVG `<filter>` element, `backdrop-filter`, the component,
chromatic aberration (that is 004). Also out of scope: the `'turbulence'` mode ‚Äî
it is procedural (`feTurbulence` built inline in the SVG filter, plan 004) and is
NOT a data-URL map, so `getDisplacementMap`'s parameter type excludes it
(`Exclude<DisplacementMode,'turbulence'>`) ‚Äî the call is prevented at the type
level rather than handled defensively at runtime. Do not copy
upstream's base64 image strings ‚Äî generate maps from code. Keep this module free
of React.
